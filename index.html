HRMS 项目中的
AI 工程化提效实践

基于 Claude Code、MCP、SKILL、项目知识库的前端开发工作流建设

HRMS AI Efficiency Report

各位好，今天我汇报的主题是 HRMS 项目中的 AI 工程化提效实践。

我想先说明一下，这次汇报不是介绍某一个 AI 工具，也不是说 AI 帮我写了几段代码。重点是我在 HRMS 项目里，把接口文档、页面映射、项目规范和 Claude Code 串成了一套可以持续运行的工作流。

这套工作流的目标，是让 AI 不再只靠提示词猜代码，而是能读取项目真实接口、页面关系和开发规范，再在这些约束下辅助前端开发。今天我会先讲业务价值，再讲技术架构，最后讲边界和后续规划。

01 · Background

前端开发的现实问题：重复、分散、易错

页面多

列表、表单、详情、弹窗结构相似，但每个页面字段和接口不同。

接口多

HRMS + basic 模块接口数量大，手动查找和维护成本高。

文档分散

Torna 管接口定义，Confluence 管页面接口关系，前端代码另有一套 API 文件。

AI 易失控

没有项目上下文时，AI 容易自编字段、乱用组件、风格不统一。

问题切入：不是不会写，而是重复工作占比高

先看背景。HRMS 这类后台系统的特点是页面多、接口多、字段多。前端开发里有很多工作不是技术难度特别高，而是非常重复，比如查接口、对字段、写 API 函数、搭列表页结构、补弹窗表单。

同时，信息来源比较分散。接口定义在 Torna，页面和接口的关系在 Confluence，前端实际调用又在 src/api 里面。开发一个页面之前，经常需要在几个地方来回确认。

如果直接让 AI 写页面，它并不知道我们项目的真实字段和规范，就容易出现自编字段、组件用错、分页规则不一致这些问题。所以我的切入点是：先把这些分散信息变成 AI 可以读取的项目知识，再让 AI 参与开发。

02 · Goal

目标：从“单次问答”升级为工程化 AI 工作流

普通 AI 使用 项目级 AI 工作流
临时问一句，生成一段代码 固定触发词和可重复执行流程
AI 靠上下文猜字段 AI 读取真实接口字典和页面映射
输出风格不稳定 CLAUDE.md + rules + templates 约束
个人经验不可复制 脚本、MCP、文档沉淀为项目资产
核心观点：不是“会用 AI”，而是“把 AI 接入研发链路”

这一页是我这次实践的核心目标。普通 AI 使用通常是临时问一句，让它生成一段代码。这种方式短期有用，但输出不稳定，也很依赖个人经验。

我希望做的是项目级工作流：用固定入口触发流程，用脚本做确定性自动化，用 MCP 连接外部系统，用 docs 沉淀项目知识，用 CLAUDE.md 约束技术栈和编码规范。

这样 AI 不只是一个聊天工具，而是被接入到真实研发链路中，能持续读取最新接口和项目规范，辅助标准化页面开发。

03 · Architecture

整体架构：交互层 · 工具层 · 知识层

交互层

Claude Code / 中文触发词：更新 API、更新页面映射、更新全部、按设计稿做页面。

工具层

Scripts + MCP + Agent + Skill：Torna 拉取、Confluence 抓取、API 生成、页面映射、检索、校验、验证。

知识层

docs + CLAUDE.md：接口参数、页面接口映射、项目规范、列表页模板、技术栈约束。

AI 要稳定参与项目，不能只靠模型本身，必须给它工具和知识。

三层模型用于解释整体工作流

这一页是整体架构。我把它分成三层。

最上面是交互层，也就是我们日常跟 Claude Code 协作的方式。开发者不用记复杂命令，可以直接说“更新 API”、“更新页面映射”、“按设计稿做页面”。

中间是工具层。这里包括项目脚本、MCP、Agent 和 Skill。脚本负责确定性的生成流程，MCP 负责连接 Confluence 和 Figma 这类外部系统，Agent 负责检索和校验这种子任务，Skill 负责通用能力。

最下面是知识层，也就是 AI 真正需要读取的项目资料。没有这一层，AI 写代码就容易靠猜。有了这一层，它才能基于项目真实数据工作。

04 · Tooling Split

工具分工：不同能力解决不同问题

要解决的问题 使用的能力 在本项目中的作用
固定流程要稳定重复执行 Scripts 拉取接口、解析文档、生成 API 和映射文件。
Claude 要访问内部系统 MCP 连接 Confluence / Figma，让 AI 能调用外部数据源。
局部任务复杂且耗上下文 Agent 负责 API 检索、页面骨架提取、规范校验。
高频 AI 工作流要标准化 Project Skills 沉淀页面生成、API 更新、菜单生成等固定流程。
项目规则需要持久化 CLAUDE.md 沉淀技术栈、触发词、代码规范和工作流入口。
不是工具堆砌，而是每类能力对应一个明确问题

这一页我不想把它讲成技术名词解释，而是讲这些工具为什么需要同时存在。

固定、可重复的流程，比如接口拉取和文件生成，适合用 Scripts。Claude 要访问内部 Confluence 或 Figma，适合用 MCP。像 API 检索、页面骨架提取、规范校验这种局部复杂任务，适合交给 Agent。

Skill 这里主要指项目级 Skill。比如 design-to-code 规定了从截图、Figma、参考页面到 Vue 页面生成的步骤；generate-api 和 update-api 负责接口更新和全部同步；generate-menu-import 负责菜单导入 JSON。

最后，CLAUDE.md 负责把项目技术栈、触发词、代码规范和工作流入口持久化。这样看，这不是工具堆砌，而是每类能力解决一个明确问题。

05 · Implementation Logic

AI 提效落地思路：先补上下文，再工具化，最后受控生成

1. 补上下文

把接口字段、页面接口关系、项目规范、页面模板，变成 AI 可以读取的资料。

2. 工具化

把接口同步、页面映射同步、字段检索、规范校验，变成可重复执行的流程。

3. 受控生成

AI 先查资料、再生成代码；字段映射、业务逻辑和最终质量仍由开发确认。

接口上下文有哪些接口、字段是什么
页面上下文页面该用哪些接口
规范上下文组件、表格、按钮、分页
参考上下文类似页面怎么实现
受控输出生成初稿，人来把关
后面几页按这个思路展开：先讲接口上下文，再讲页面映射和受控生成

前面讲了问题、目标、架构和几个技术概念。这里我补一层落地思路，否则后面直接讲接口链路会有点突然。

我没有直接把目标定成“让 AI 自动写页面”，因为这样风险很大。AI 如果没有项目上下文，就会自编字段、乱用组件，甚至写出看起来合理但实际不可用的代码。

所以我的思路分三步：第一，先补上下文，把接口字段、页面接口关系、项目规范和页面模板都变成 AI 可以读取的资料；第二，把重复流程工具化，比如接口同步、页面映射同步、字段检索；第三，再让 AI 在这些上下文和工具约束下生成代码，并由开发确认字段、业务逻辑和最终质量。

接下来几页我会按这个思路展开。先看第一块：接口上下文是怎么自动化的。

06 · API Automation

接口链路：把“查文档、写接口、对字段”变成一键同步

1. 接口来源

HRMS 从 Torna 自动拉取；basic 保留一个手动导出文件，作为公共模块补充。

2. 自动生成

统一生成前端请求函数、接口参数字典和 API 索引，避免人工维护多份资料。

3. AI 可读取

AI 做页面时直接读取真实字段和函数路径，不再凭描述猜接口。

后端文档Torna + basic export
同步生成更新 API
前端调用src/api
字段字典api-params
AI 生成页面按真实字段输出
一句话：接口文档变成前端和 AI 都能消费的标准数据

这页我把接口链路讲得更直白一点。以前开发页面时，前端要先去 Torna 查接口，再手动确认字段，再写 API 函数。这些工作很重复，也容易因为文档更新不同步出现问题。

现在这条链路分三步：第一，HRMS 核心接口从 Torna 自动拉取；basic 作为公共模块，保留一个手动导出文件。第二，统一生成前端请求函数、接口参数字典和 API 索引。第三，AI 做页面时可以直接读取这些结构化结果。

这部分对应的是我沉淀的 generate-api 项目级 Skill。它把“更新 API”这件事固定成标准流程：先拉 Torna，再合并 basic，最后生成前端 API、参数字典和索引。

这里也可以把技术落地点讲清楚：Skill 负责定义流程和入口，fetch-torna-export.js 负责通过 Torna API 自动拉取 HRMS 接口文档，generate-api.js --hybrid 负责把 Torna JSON 和 basic 的手动 export 合并，生成最终产物。

也就是说，这里不是让 AI 临时读一份文档再写代码，而是把“拉接口、解析接口、生成代码和字段字典”沉淀成一个项目级 Skill 加脚本的固定工作流。AI 后续再读取这些产物辅助页面开发。

所以它的价值不是多一个脚本，而是把接口文档变成前端和 AI 都能消费的标准数据。后端只要维护好 Torna，前端就可以持续同步。

07 · Page Mapping

仅知道“有哪些接口”还不够，还要知道页面用哪些接口

接口字典回答

这个接口有哪些字段？

来源：Torna / basic export 生成的 api-params

页面映射回答

这个页面该用哪些接口？

来源：Confluence 页面接口表生成的 page-api-map

MCP 同步

抓取 Confluence 表格，解析菜单 / 功能 / 接口，生成页面到 API 的结构化映射。

匹配统计 + Diff

统计页面数、API 数、精确/模糊/兜底/未匹配；Diff 展示新增、移除、修改。

先定位页面接口，再读取接口字段；匹配统计看准不准，Diff 看变了什么

这一页是为了讲清楚为什么除了 API 字典，还需要页面映射。

API 字典回答的是“某个接口有哪些字段”。但做页面时还有一个更前置的问题：这个页面到底应该用哪些接口？这个关系通常在 Confluence 的页面接口表里。

所以这里通过 MCP 把 Confluence 表格同步下来，生成 page-api-map。AI 做页面时，先通过页面映射定位候选接口，再回到 api-params 读取字段。这样比单纯让 AI 在几百个接口里搜索要更准确。

下半部分是这一步的技术补充：MCP 会抓取 Confluence 表格，解析菜单、功能、接口名称、接口地址等列，再生成结构化页面映射。

同步后还会输出匹配统计和 Diff 报告。匹配统计会展示抓取页面数、提取 API 数、名称精确匹配、名称模糊匹配、URL 精确匹配、关键词兜底、模块兜底和未匹配数量。这个统计用于判断生成得准不准。

Diff 报告则关注变了什么，比如页面/API 的新增、移除和修改。这样页面映射不是黑盒生成文件，而是可检查、可追踪的同步流程。

10 · Page Generation

最终效果：AI 做页面时，走的是受控流程

看页面截图 / Figma / 参考页面
定接口先查页面映射
对字段再查接口参数
套规范Vue2 / BaseVxeTable / 项目规则
菜单配置新页面生成菜单导入 JSON
人确认字段映射和业务逻辑仍由开发把关
AI 负责重复和结构化工作，人负责业务判断和最终质量

前面几页讲了接口字典和页面映射，这一页把它们放回页面生成场景里。

AI 做页面时，第一步是看页面结构，比如截图、Figma 或参考页面。第二步不是立刻写代码，而是先通过 page-api-map 定位这个页面应该用哪些接口。第三步再通过 api-params 对字段。第四步才是按项目规范生成 Vue 页面。

这部分对应的是 design-to-code 项目级 Skill。它把页面生成拆成固定步骤：识别输入、查 API、提取参考页面骨架、生成字段映射、字段确认后再写代码，最后做规范校验。

另外，新页面不只是生成 Vue 文件，还需要能被系统菜单挂载。所以我也沉淀了 generate-menu-import 项目级 Skill：它会根据已有 Vue 页面、页面标题、路径和当前菜单层级，生成基础服务可导入的菜单 JSON，减少手工配置菜单层级和权限字段的错误。

右侧展示的是实际执行过程的抽象。它不是直接开始写 Vue 页面，而是先通过 Figma MCP 获取设计稿节点数据；再用子 Agent 提取参考页面骨架；再用 API Agent 检索候选接口。这样视觉上下文、参考上下文和接口上下文都准备好之后，才进入字段确认和代码生成。

这里用子 Agent 的一个重要原因，是把查资料、读文档、提骨架这类耗上下文的任务从主 Claude 中拆出去。主 Claude 不需要塞进大量文档细节，只接收结构化结论，再负责整体决策和代码整合。

这里的核心是受控。我们不让 AI 猜接口，不让 AI 乱写组件，也不让 AI 自动替代业务判断。AI 负责重复和结构化的部分，开发负责字段确认、业务逻辑和最终质量。

09 · Menu Import

菜单配置也纳入工作流：从手工拼参数到直接导入 JSON

原来的痛点新页面不只是写 Vue，还要配置路由名称、URL、组件路径、重定向、父级层级等参数。
容易出错name 唯一性、Layout / Root / 叶子节点关系、activeMenu、hidden、redirect 等都需要人工判断。
现在的方式根据实际菜单树和页面路径，按预置详细规则生成基础服务可直接导入的标准 JSON。
规则约束Layout / Root / 叶子节点、redirect、hidden、activeMenu、唯一 name 等都按规则推导。
对应能力generate-menu-import 项目级 Skill：读取页面信息 + 当前菜单关系，输出完整导入结构。
页面生成不仅是 Vue 文件，也包括能落到系统菜单里的配置结果

这里我补充一个之前容易被忽略，但实际开发里很花时间的环节：菜单配置。

新页面写完 Vue 文件以后，并不代表就能在系统里访问。还需要在基础服务里配置菜单，包括路由 name、访问 URL、组件路径、父级菜单层级、重定向、是否隐藏、activeMenu、按钮权限等参数。

这些参数如果手工配，既繁琐又容易错。比如 name 要唯一，Layout 下面不能直接挂页面，中间要有 Root，叶子节点要对应真实页面路径，详情页还要考虑高亮父菜单。

所以我把这一步也沉淀成了 generate-menu-import 项目级 Skill。它会根据已有 Vue 页面、页面标题、路径和当前菜单树，按预置的详细规则推导出基础服务可直接导入的标准菜单 JSON。比如 Layout 下只能挂 Root，页面必须挂在 Root 下，叶子节点 hidden、父级 redirect、详情页 activeMenu、name 唯一性等，都按规则处理。

这一步的价值是把“页面写完以后还要手工想菜单怎么配”的工作，也纳入 AI 工作流。这样新页面从代码到菜单配置可以形成闭环，减少人工拼参数和层级配置错误。

11 · Realistic Boundary

AI 提效不是“自动完成项目”，而是减少重复劳动

AI 适合提效的部分

  • 查接口、对字段、定位相似页面
  • 生成 API 文件和参数字典
  • 搭标准列表页 / 表单页骨架
  • 按项目规范补基础代码
  • 减少重复性的文档查找和复制粘贴

仍然需要开发投入的部分

  • 业务规则理解和需求澄清
  • 复杂交互、流程状态、权限边界判断
  • 前后端联调和异常场景处理
  • 测试验收、问题修复、需求变更沟通
  • 最终代码质量和交付责任
AI 缩短的是重复编码和资料查找时间，不是消除业务实现和质量交付成本。
预期管理：AI 是提效工具，不是项目自动完成器

这里我想主动讲一下 AI 提效的真实边界，避免大家产生一个误解：有了这套工作流，页面就可以自动完成，项目就会很快结束。

实际不是这样。AI 在这个项目里最适合提效的，是标准化、重复性强、规则比较明确的部分，比如查接口、对字段、生成 API 文件、搭标准页面骨架、按规范补基础代码。

但是还有很多事情仍然需要开发投入时间，比如业务规则理解、需求澄清、前后端联调、复杂交互、异常场景处理、测试验收和需求变更沟通。

所以我对这套工作流的定位不是“让 AI 替代开发完成项目”，而是“让 AI 减少重复劳动，让开发把时间更多放在业务判断和质量交付上”。这也是我前面一直强调受控生成和人工确认的原因。

已形成的价值：从个人使用 AI 到项目级资产

效率

减少查 Torna、对字段、写 API、搭页面骨架等重复工作。

质量

字段来自真实接口，页面接口关系来自 Confluence，输出可追溯。

协作

后端文档质量、Confluence 表格质量可以直接转化为前端效率。

沉淀

scripts、MCP、CLAUDE.md、docs 形成可维护、可持续迭代的项目资产。

这不是一次“AI 写代码尝试”，而是把 AI 接入 HRMS 研发链路。
价值表达：效率、质量、协作、资产

这一页总结已经形成的价值。第一是效率，减少大量重复性的文档查找和代码骨架工作。

第二是质量。字段来自真实 API 文档，页面接口关系来自 Confluence，项目规范写在 CLAUDE.md 和 docs 里，所以生成结果更可控。

第三是协作。后端只要维护好 Torna，页面接口关系维护好 Confluence，前端就可以自动同步。这实际上把文档质量和开发效率打通了。

第四是资产沉淀。这些不是一次性的提示词，而是项目里的脚本、MCP、文档和规则，后续可以继续维护，并随着 HRMS 页面开发不断迭代。

14 · Risk Control

风险控制：文档、凭证、生成结果都要可追踪

风险点 控制方式
接口文档不准确 Torna 作为源头，生成后仍由前端 review 关键字段和调用方式。
页面接口关系不准确 Confluence 同步输出匹配统计和 Diff，变化可追踪、可回看。

第 13 页讲预期边界，这一页讲流程风险如何控制

前面一页讲的是 AI 提效的真实边界，这一页讲风险控制。重点不是再说 AI 不能替代开发，而是说明这套流程如何做到可追踪、可排查。

第一类风险是文档准确性。接口文档以 Torna 为源头，页面接口关系以 Confluence 为源头，但生成后仍然需要前端 review 关键字段和调用方式。

第二类风险是同步和生成过程。Confluence 同步会输出匹配统计和 Diff，中间也保留 raw 或 JSON 产物，所以问题可以定位到拉取、解析、匹配或生成阶段。

第三类是安全风险。Token 和 Cookie 都放在用户本地配置目录，不进 Git，也不进入对话上下文。整体目标是让 AI 工作流可用，同时保证过程可追踪、凭证可控、结果可审核。
